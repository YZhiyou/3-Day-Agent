## 学习路径：如何在 3 天内用 AI 掌握 LangChain Agent 开发？

### 1. 传统学习已跟不上 AI 框架的迭代

LangChain、LlamaIndex、MCP、Skill…新名词每周都在涌现，接口几个月就可能面目全非。传统视频、书籍等学习方式面临三重困境：

- **更新速度脱节**：你刚看完教程，`AgentExecutor` 的导入路径已经变了。
- **信息洪流无序**：官方文档追求完整，却不负责“从哪里开始”。
- **0 到 1 的鸿沟太难跨**：买本书？查文档？你连第一个能跑的链都搭不起来。

我就是这样一个零基础的学习者。但这一次，我决定用 **AI 本身** 来解决 AI 学习的问题。

### 2. 核心理念：用 AI 学习 AI

如果你仔细观察，会发现大多数 AI 学习痛点都指向同一个解：**把 AI 变成你的导师、同桌和代码副手**。

我设计并实践了一套 **“三窗口协同学习法”**，突破了单对话上下文限制，实现：

- **知识获取** 与 **动手实战** 实时交替。
- 在极高的信息密度下，**不走神、不迷路、不拖延**。
- 仅用 **3 天** 就完成从 LangChain 扫盲到交付 **全栈智能体应用** 的全过程。

### 3. 三窗口协同学习法详解

```
                   ┌─────────────────┐
                   │   你（主控大脑）  │
                   └───────┬─────────┘
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
┌─────────────────┐ ┌───────────┐ ┌─────────────────┐
│  主控调度室      │ │ 知识专题   │ │  代码实战        │
│  (1个窗口)       │ │ 研讨室     │ │  训练营          │
│                 │ │ (多窗口)   │ │  (1个窗口+AI IDE)│
│ - 制定学习路径   │ │           │ │                 │
│ - 生成研究提示词 │ │ - 概念讲解 │ │ - 模块编码       │
│ - 审核整合成果   │ │ - 交互问答 │ │ - 调试优化       │
│                 │ │ - 生成笔记 │ │ - 版本适配       │
└─────────────────┘ └───────────┘ └─────────────────┘
```

#### 🕹️ 主控调度室（1 个持续单窗口）

学习的“大脑”。从不与 AI 纠缠具体代码，只做三件事：

1. 制定全局学习计划；
2. 把大任务拆解为可独立执行的子任务，并生成**结构化的启动提示词**；
3. 审查每个窗口产出的“知识资产”（笔记、代码），整合进入下个阶段。

#### 📚 知识专题研讨室（多个短对话窗口）

每个窗口只聚焦一个模块（Prompt 工程、RAG、Agent…），用 **“规划先行，分步教学”** 的提示词强制 AI 逐段讲解，讲透一个点再进入下一个。**从根本上杜绝一次性长篇灌输。**

学完一个模块后，AI 必须自动输出一份《学习笔记摘要》，这就是跨窗口传递上下文的“知识资产”。

#### ⚔️ 代码实战训练营（专属窗口 + AI IDE）

本项目的代码实战完全采用 **AI IDE** 负责编码与调试，**所有报错信息都在 IDE 内闭环解决，绝不贴回知识研讨窗口造成上下文污染**。

- 对话 AI（DeepSeek）负责：**“做什么”和“为什么这么做”** → 提供模块设计思路、架构解释。
- AI IDE 负责：**“具体怎么写”和“怎么调”** → 根据功能描述生成代码、运行、发现报错、在 IDE 内反复修正，直到跑通。

> **核心洞察**：LangChain 等框架接口变化极快，**纠结具体代码写法意义不大**。真正需要掌握的是 **业务逻辑**：Agent 的输入→LLM→输出，加上历史记录就有了记忆，加上检索就有了知识库，加上工具就有了各种能力。RAG 的流程（加载→分块→向量化→存储→检索→增强→生成）同样是一种稳定的业务逻辑，远比背 API 重要。

这种分工让我们的 **学习生产力爆炸性提升**：两天内完成全部核心理论学习，第三天直接交付全栈应用。

### 4. 3 天速成路线图

| 时间 | 学习主题 | 核心产出 | 关键概念 |
|------|---------|---------|---------|
| **Day 1** | 基础认知 + 核心组件 | 跑通第一个链，模型能按 pydantic 格式输出 | LCEL, PromptTemplate, PydanticParser |
| **Day 2** | RAG + Agent 智能体 | 构建知识库问答，Agent 学会调用工具 | Chroma, Embedding, ReAct, Tools |
| **Day 3** | 记忆管理 + Plan-Act + 全栈实战 | 交付带前端的多轮对话知识库助手，支持复杂任务自动分解与执行计划可视化 | Memory, LangGraph, Plan-Act, Streamlit, SqliteSaver |

> **Day 2 结束时，所有 LangChain 核心理论知识已全部结束。Day 3 完全用于项目落地**，最终产出一个功能完整、架构清晰的 **Personal Knowledge Base Agent**。

### 5. 本项目（3-Day-Agent）能力速览

这正是 Day 3 交付的 Agent 所具备的全部能力：

| 能力 | 说明 |
|------|------|
| **个人知识库检索** | 调用 `retrieve_documents(query)` 搜索本地 Chroma 向量库中的 PDF、Markdown、TXT；采用 **Parent-Child** 分块策略：子块（chunk_size=300）嵌入 Chroma 负责精确召回，父块（chunk_size=1000）经 `JsonDocstore` 持久化保障完整上下文返回。检索 pipeline：子块语义召回 → 返回父块 → BM25 关键词精排 → RRF 融合 → DashScope Rerank (`qwen3-vl-rerank`) 最终精排返回 top-5。尤其擅长缩写与术语的精确锁定。 |
| **长期用户记忆** | 调用 `save_user_info(key, value)` / `get_user_info(key)` 读写用户 JSON 文件，持久化姓名、偏好等。 |
| **联网实时搜索** | 通过 `web_search(query)`（DuckDuckGo）获取实时信息并附来源链接。 |
| **短期会话记忆** | 利用 LangGraph + SqliteSaver 持久化对话，支持跨重启的连续对话，session 隔离。 |
| **复杂任务 Plan-Act 分解** | Agent 自动判断任务复杂度：复杂任务生成多步计划、逐条执行、动态检查进度、偏离时自动重规划（最多 3 次），最终汇总生成回答。简单任务直接走原 ReAct 流程。 |

**技术架构分层：**

```
┌─────────────────────────────────────────────────────────┐
│              Streamlit WebUI / CLI                      │  用户界面层
│              └─ 执行计划卡片（复杂任务时侧边栏显示）         │
├─────────────────────────────────────────────────────────┤
│              agent.py (create_agent)                    │  智能体调度层
│   ┌─────────────────────────────────────────────────┐   │
│   │  Plan-Act 外层 StateGraph                        │   │
│   │  ├─ classify_complexity → 判断简单/复杂           │   │
│   │  ├─ 简单 → react_agent (原 ReAct 子图)           │   │
│   │  └─ 复杂 → generate_plan → execute_step          │   │
│   │            ↔ check_progress (偏离则 replan)       │   │
│   │            → summarize_results                   │   │
│   │  + SqliteSaver 记忆持久化 (checkpoints.db)        │   │
│   └─────────────────────────────────────────────────┘   │
├─────────────────────────────────────────────────────────┤
│  tools.py                                               │  工具层
│  ├─ retrieve_documents (知识库检索)                      │
│  ├─ save_user_info / get_user_info                      │
│  └─ web_search (联网搜索)                                │
├─────────────────────────────────────────────────────────┤
│  retriever.py / reranker.py / kb_manager.py             │  服务层
│  ├─ Parent-Child 分块（子块嵌入 / 父块上下文持久化）        │
│  ├─ Chroma 向量检索（子块语义粗筛召回 top-k）               │
│  ├─ BM25 关键词精排（在父块候选集上做字面匹配）              │
│  ├─ RRF 倒数排名融合（语义+关键词双路合并）                 │
│  ├─ DashScope Rerank 语义重排序（精排返回 top-n）          │
│  └─ 知识库增删重建                                        │
├─────────────────────────────────────────────────────────┤
│  vector_store.py                                        │  数据持久层
│  ├─ DashScope Embedding 生成                             │
│  └─ Chroma 向量库管理                                    │
├─────────────────────────────────────────────────────────┤
│  memory_manager.py                                      │  记忆层
│  ├─ 短期会话历史 (SQLite checkpoints)                    │
│  └─ 长期用户记忆 (JSON)                                  │
└─────────────────────────────────────────────────────────┘
```

Agent 的系统提示词中内置了决策优先级：
1. 询问**个人信息** → 先查长期记忆
2. 询问**知识点/参考资料** → 先检索本地知识库
3. 涉及**实时动态** → 则进行联网搜索
4. 绝不编造信息，知识库无结果时诚实告知

**Plan-Act 模式决策流程**：
<img width="3061" height="1584" alt="image" src="https://github.com/user-attachments/assets/0b4550df-c598-482b-8792-fd9d3a1955f1" />

- 用户输入首先经过 `classify_complexity` 判断复杂度
- **简单任务**（闲聊、单次检索、单步读写）→ 直接由原 ReAct Agent 处理
- **复杂任务**（≥2 个工具、多步依赖）→ 自动生成执行计划 → 逐条执行 → 检查进度 → 偏离时自动重规划（最多 3 次）→ 汇总生成最终回答

### 6. 学习者必读：几个改变我认知的事实

- **AI 比你更会向 AI 提问**：不知道如何让 AI 教好你？直接让 AI 为你写“教自己 LangChain”的提示词。
- **执行，是 AI 时代最重要的技能**：动手做，用 AI IDE 加速试错，不要等“准备好”。我的整个学习过程只用了 3 天，不是因为聪明，而是因为立刻启动、不纠结细节。
- **框架会死，逻辑永生**：明天 `langchain` 可能又会换个导入方式，但 RAG 的数据流、Agent 的 ReAct 循环、记忆管理的分层思想不会变。抓住本质，你就能在任何框架间自由迁移。
