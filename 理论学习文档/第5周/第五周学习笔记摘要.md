## 📘 第五周学习笔记摘要：智能体核心——工具与推理

### 一、ReAct 循环图解

Agent 的决策引擎是 **ReAct**（Reasoning + Acting）循环，它让模型在**思考**和**行动**之间交替，直至得出结论。

```
用户提问
   │
   ▼
┌──────────────────────────────────────────────────┐
│  ┌──────────┐     ┌──────────┐     ┌──────────┐  │
│  │ Thought  │────▶│  Action  │────▶│Observation│  │
│  │ (思考)   │     │ (调用工具)│     │ (工具结果) │  │
│  └──────────┘     └──────────┘     └──────────┘  │
│        ▲                                    │     │
│        └────────── 信息不足则继续 ──────────┘     │
│                                                  │
│  当模型判定信息足够时，跳出循环，生成最终答案    │
└──────────────────────────────────────────────────┘
   │
   ▼
Final Answer（自然语言回复给用户）
```

- **Thought**：模型分析当前问题，决定是否需要工具、用哪个工具、传什么参数。
- **Action**：实际调用某个工具（如 `get_weather(city="北京")`）。
- **Observation**：工具返回的原始数据（例如 `"北京：晴天，气温18℃"`）。
- **结束条件**：模型评估已有信息已足够回答用户，则输出 `Final Answer`；否则继续下一轮循环。执行器还会通过 `max_iterations`（或 `recursion_limit`）强制终止超长循环。

### 二、关键代码资产

以下是一个**完整可运行**的 Agent 构建示例，包含两个自定义工具、Agent 创建、记忆增强及多轮交互（适配 LangChain ≥1.2.15 与最新导入）。

```python
import os
import math
from langchain_openai import ChatOpenAI
from langchain.agents import create_agent
from langchain_core.tools import tool
from langchain_core.messages import HumanMessage, AIMessage
from langchain_core.globals import set_verbose

# ==================== 1. 定义工具 ====================
@tool
def get_weather(city: str) -> str:
    """查询指定城市的实时天气情况。
    
    Args:
        city: 需要查询天气的城市名称（中文或英文）。
    
    Returns:
        一个字符串，描述该城市的天气状况和气温。
    """
    weather_data = {
        "北京": "晴天，气温18℃",
        "上海": "多云转阴，气温24℃",
        "广州": "雷阵雨，气温28℃",
        "深圳": "晴间多云，气温27℃"
    }
    return weather_data.get(city, f"{city}天气：局部多云，气温22℃。")

@tool
def calculator(expression: str) -> str:
    """执行数学计算。支持基本的四则运算和部分数学函数（如 math.sqrt）。
    
    Args:
        expression: 一个有效的 Python 数学表达式字符串，例如 "3 * (5 + 2)"。
    
    Returns:
        计算结果的字符串形式；如果表达式无效则返回错误信息。
    """
    try:
        allowed_names = {k: v for k, v in math.__dict__.items() if not k.startswith("__")}
        result = eval(expression, {"__builtins__": {}}, allowed_names)
        return str(result)
    except Exception as e:
        return f"计算错误：{e}"

tools = [get_weather, calculator]

# ==================== 2. 初始化模型与 Agent ====================
llm = ChatOpenAI(model="gpt-3.5-turbo", temperature=0)
agent = create_agent(llm, tools)          # 新版标准，返回 Runnable
set_verbose(True)                         # 开启 verbose 观察工具调用

# ==================== 3. 为 Agent 注入记忆（简易版） ====================
history = []  # 储存多轮对话历史

def chat_with_memory(user_input: str) -> str:
    """带记忆的对话函数：将历史与新问题拼接后传给 Agent。"""
    global history
    messages = history + [HumanMessage(content=user_input)]
    result = agent.invoke({"messages": messages})
    ai_response = result["messages"][-1].content

    # 更新对话历史
    history.append(HumanMessage(content=user_input))
    history.append(AIMessage(content=ai_response))
    return ai_response

# ==================== 4. 多轮交互演示 ====================
if __name__ == "__main__":
    print("🤖 带记忆的 Agent 已启动 (输入 'quit' 退出)\n")
    while True:
        user_input = input("你：")
        if user_input.lower() in ["quit", "exit"]:
            break
        response = chat_with_memory(user_input)
        print(f"Agent：{response}\n")
```

### 三、调试心法（通过 Verbose 日志分析 Agent 行为）

在开发过程中，你将通过控制台输出检查 Agent 的每一步决策。以下是两条实用技巧：

#### 1. 如何判断工具描述（docstring）是否写得好？

**现象**：Verbose 日志显示 Agent 调用了与问题完全不相关的工具，或者使用了错误的参数名。

**排查步骤**：
- 查看日志中的 `Action Input` 字段，看参数名是否符合你的工具 docstring 中 `Args` 的定义。
- 如果 Agent 没调用工具而直接瞎编答案，可能是 docstring 没清楚说明“应该在何时使用该工具”。
- **修正方法**：在 docstring 开头用一句话明确工具的使用场景，例如 *“当需要获取城市天气时使用此工具”*，并确保参数描述精确。

#### 2. 如何诊断 Agent 无限循环或提前终止？

**现象**：Agent 反复调用同一个工具，或明明信息不足却直接给出了 `Final Answer`。

**排查步骤**：
- 观察 Verbose 日志中每一轮的 `Observation` 内容。如果工具返回的信息模糊或有歧义，Agent 可能陷入自我怀疑的循环。
- 如果 `Observation` 里包含错误信息（如 `"计算错误：..."`），Agent 可能无法正确处理，此时应修改工具让错误信息更明确，或启用 `handle_parsing_errors=True`（旧版 AgentExecutor）让模型尝试自动修复。
- 若 Agent 过早结束，检查是否将关键数据写在了 `Observation` 之外，或者提示词没有强调“必须基于观察事实回答”。
- 使用 `max_iterations`（或 `recursion_limit`）作为安全阀，并在日志中留意达到限制时的 `early_stopping_method` 行为。
