## 🔍 一、查看检索到的源文档（`return_source_documents`）

当你发现模型的回答不够准确或“幻觉”较多时，第一步是**查看检索器返回了哪些文档片段**。这能帮你判断问题是出在检索环节（没召回相关内容）还是生成环节（模型理解有误）。

### 1.1 在 RAG 链中返回源文档

修改链的输出，同时包含答案和源文档：

```python
from langchain_core.runnables import RunnableParallel, RunnablePassthrough, RunnableLambda

# 定义检索函数（返回文档对象列表）
def retrieve_docs(question):
    docs = retriever.invoke(question)
    return docs

# 定义生成答案的函数（接收 context 和 question）
def generate_answer(context, question):
    formatted_prompt = prompt.format_messages(context=context, question=question)
    response = llm.invoke(formatted_prompt)
    return response.content

# 构建并行执行的两个分支：
# - 分支1：生成答案（需要先获取 context）
# - 分支2：直接返回源文档
rag_chain_with_source = (
    # 先准备好 context 和 source_documents
    RunnablePassthrough.assign(
        context=lambda x: "\n\n".join([d.page_content for d in retrieve_docs(x["question"])]),
        source_documents=lambda x: retrieve_docs(x["question"])
    )
    # 然后用 RunnableParallel 同时计算答案和保留 source_documents
    | RunnableParallel(
        answer=lambda x: generate_answer(x["context"], x["question"]),
        source_documents=lambda x: x["source_documents"]
    )
)

# 调用
result = rag_chain_with_source.invoke({"question": "文档中的核心观点是什么？"})
print("答案:", result["answer"])
print("\n--- 使用的源文档片段 ---")
for i, doc in enumerate(result["source_documents"]):
    print(f"片段 {i+1} (来自 {doc.metadata.get('source', 'unknown')} 页 {doc.metadata.get('page', '?')}):")
    print(doc.page_content[:300])
    print("-" * 40)
```

> ⚠️ 注意：上面的写法将 `source_documents` 作为额外键传递，但 `prompt` 只使用 `context` 和 `question`，所以不会影响答案生成。最终结果对象会包含 `source_documents` 字段。

### 1.2 更简洁的调试方式（不修改链）

直接使用 `retriever` 查看检索结果：

```python
question = "什么是RAG？"
retrieved = retriever.invoke(question)
for i, doc in enumerate(retrieved):
    print(f"Top {i+1}: {doc.page_content[:200]}...")
    print(f"Metadata: {doc.metadata}\n")
```

这样你可以快速判断检索质量。

## ⚙️ 二、调整分割参数（`chunk_size` / `chunk_overlap`）

分割参数直接影响检索颗粒度和上下文完整性。

| 问题现象                         | 可能原因                          | 调优方向                              |
| -------------------------------- | --------------------------------- | ------------------------------------- |
| 答案过于零碎、缺乏上下文         | `chunk_size` 太小                 | 增大 `chunk_size`（如从 500 → 800）   |
| 答案包含不相关内容               | `chunk_size` 太大，单个块信息过杂 | 减小 `chunk_size`（如从 500 → 300）   |
| 关键信息正好落在块边界，检索不到 | `chunk_overlap` 太小              | 增大 `chunk_overlap`（如从 50 → 100） |
| 存储和检索速度慢                 | 总块数过多                        | 增大 `chunk_size`，减少块数量         |

### 调优实验代码模板

```python
chunk_sizes = [300, 500, 800]
overlaps = [30, 50, 100]

for size in chunk_sizes:
    for overlap in overlaps:
        splitter = RecursiveCharacterTextSplitter(
            chunk_size=size, chunk_overlap=overlap
        )
        chunks = splitter.split_documents(pages)
        print(f"size={size}, overlap={overlap} -> {len(chunks)} chunks")
        # 可以进一步测试几个问题的检索质量
```

建议：先用固定几个问题手动评估，找到最佳平衡点。

## 🎯 三、调整检索参数（`k` 与检索策略）

### 3.1 修改检索数量 `k`

`retriever` 默认返回 `k=4` 个文档块。你可以通过 `search_kwargs` 调整：

```python
retriever = vectordb.as_retriever(search_kwargs={"k": 3})   # 返回3个块
```

- **k 太小**：可能遗漏关键信息。
- **k 太大**：引入噪声，增加 token 消耗，且可能分散模型注意力。

建议从 `k=3` 或 `k=4` 开始，根据文档长度和问题复杂度调整。

### 3.2 切换检索类型（可选）

Chroma 支持 `similarity`（默认）和 `mmr`（最大边际相关性）两种检索策略。MMR 能减少返回结果之间的冗余。

```python
retriever = vectordb.as_retriever(
    search_type="mmr",
    search_kwargs={"k": 4, "fetch_k": 10}
)
# fetch_k 表示先检索10个候选，再用 MMR 筛选出4个
```

当你发现返回的多个片段内容高度重复时，可以尝试 MMR。

## 📝 五、调优备忘（实用技巧）

### 技巧1：根据文档类型选择分块策略

- **技术文档/代码**：使用 `RecursiveCharacterTextSplitter` 并加入代码分隔符（如 `"\n\n", "\n", " ", ""`），或者使用 `LanguageParser` 针对特定语言。
- **连续叙述的文章**：较大的 `chunk_size`（800-1200）和适度重叠（100-200）效果好。
- **问答对格式**：保持每个问答对在一个块内，可以自定义分割逻辑。

### 技巧2：处理检索到的多个文档片段

- **重排序**：在送入 LLM 前，用一个轻量级模型（如 `CrossEncoder`）对检索结果重新打分排序，让最相关的片段排在前面。
- **压缩**：如果总长度超过模型上下文窗口，可以使用 `LLMChainExtractor` 或 `ContextualCompressionRetriever` 提取最关键的句子。

### 技巧3：增加“引用来源”输出

在 prompt 中要求模型输出答案时标明参考的片段编号，提高可信度：

```python
template = """...（上下文）...

请回答用户问题，并在答案末尾标注引用了哪些片段（例如 [片段1][片段3]）。"""
```
