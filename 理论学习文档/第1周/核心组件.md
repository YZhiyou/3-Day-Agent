## Agents（智能体）

根据你提供的 LangChain 官方文档，我来为你详细解析一下 **Agents（智能体）** 这个核心组件。简单来说，Agent 是一个将语言模型的**推理能力**与**具体行动（工具）** 结合起来的系统，它能自主思考、决策并迭代执行任务，直至达成目标。

### 核心机制：ReAct 循环
Agent 的工作流遵循 **ReAct（推理 + 行动）** 模式，这是一个持续的循环过程：
1.  **推理**：Agent 接收任务，思考下一步需要做什么。
2.  **行动**：根据推理，决定调用哪个工具（如搜索、计算器），并执行。
3.  **观察**：接收工具执行后的返回结果。
4.  **循环**：将观察结果作为新信息，再次进入推理步骤，直到能给出最终答案。

### 快速创建 Agent
官方推荐的现代化方法是使用 `create_agent` 函数，它内置了基于 LangGraph 的生产级运行环境。

```python
from langchain.agents import create_agent
from langchain_openai import ChatOpenAI

# 1. 准备模型（Agent的大脑）
model = ChatOpenAI(model="gpt-4o")

# 2. 准备工具（Agent能用的武器）
# 工具可以是任何Python函数，需用 @tool 装饰器声明

# 3. 创建Agent
agent = create_agent(model, tools=[search, get_weather])

# 4. 运行Agent
result = agent.invoke({"messages": [{"role": "user", "content": "旧金山天气怎么样？"}]})
```

### 三大核心配置
在创建 Agent 时，除了**模型**和**工具**，还有几个关键配置能深度定制其行为：

| 配置项         | 作用                                                         | 示例场景                                                     |
| :------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **系统提示词** | 设定Agent的角色、目标和行为准则。                            | 设置为“你是一位专业的财经分析师，回答需基于数据。”           |
| **结构化输出** | 强制Agent的最终回复遵循特定的数据格式（如JSON），便于程序解析。 | 从一段文本中提取姓名、邮箱、电话，并以结构化对象返回。       |
| **中间件**     | Agent执行流程中的“钩子”，可在调用模型前后或工具执行出错时插入自定义逻辑。 | 根据对话轮次动态切换模型、统一处理工具报错、记录详细日志等。 |

### 深入理解：中间件的强大能力
中间件是 LangChain Agent 框架中极具扩展性的部分。文档中列举了两种典型用法，你可以通过简单的装饰器实现：

**1. 动态模型选择**
根据对话的复杂度自动切换模型，在成本和效果间取得平衡。
```python
from langchain.agents.middleware import wrap_model_call

@wrap_model_call
def dynamic_model_selection(request, handler):
    # 如果对话超过10轮，则使用更强的模型
    if len(request.state["messages"]) > 10:
        model = advanced_model
    else:
        model = basic_model
    # 用选定的模型继续执行
    return handler(request.override(model=model))
```

**2. 自定义工具错误处理**
当工具执行失败时，不直接抛出异常中断流程，而是返回一条友好的提示给模型，让模型可以尝试纠正或采取其他行动。
```python
from langchain.agents.middleware import wrap_tool_call
from langchain.messages import ToolMessage

@wrap_tool_call
def handle_tool_errors(request, handler):
    try:
        return handler(request)
    except Exception as e:
        # 将报错信息包装成 ToolMessage 返回给模型
        return ToolMessage(
            content=f"工具出错了，请检查输入。错误详情：{e}",
            tool_call_id=request.tool_call["id"]
        )
```

### 状态与记忆
Agent 通过一个名为 `AgentState` 的状态字典在多次交互中维持上下文。默认情况下，它会自动保存对话历史（`messages` 列表）。如果你需要记住更多信息（如用户偏好、临时变量），可以定义自定义状态。

## Models（模型）

根据你提供的 LangChain 官方文档，我来为你详细解析 **Models（模型）** 这个核心组件。简单来说，在 LangChain 中，Model 是应用的“大脑”，负责理解和生成文本。文档主要聚焦于**聊天模型（Chat Models）**，它们是构建现代 AI 应用的基础。

### 快速初始化模型
最推荐的方式是使用 `init_chat_model` 函数，它能以统一的方式接入各种模型提供商。

```python
from langchain.chat_models import init_chat_model

# 1. 最简初始化：直接指定模型名称
model = init_chat_model("gpt-5.2")

# 2. 带参数初始化：控制模型行为
model = init_chat_model(
    "claude-sonnet-4-6",
    temperature=0.7,  # 控制随机性 (0-1)
    timeout=30,       # 请求超时时间(秒)
    max_tokens=1000,  # 限制回复最大长度
    max_retries=6     # 失败重试次数
)

# 3. 调用模型
response = model.invoke("为什么鹦鹉会说话？")
```

### 三种核心调用模式

| 方法           | 特点                             | 适用场景                                       |
| :------------- | :------------------------------- | :--------------------------------------------- |
| **`invoke()`** | 一次性返回完整响应。             | 常规问答、分类、提取等非流式任务。             |
| **`stream()`** | 逐块返回生成内容，提升用户体验。 | 聊天应用、长文本生成等需要实时反馈的场景。     |
| **`batch()`**  | 并行处理多个独立请求，提升效率。 | 批量翻译、批量分类等需处理大量数据的离线任务。 |

### 赋予模型两大关键能力

Models 组件通过 `bind_tools` 和 `with_structured_output` 方法，为模型扩展了两种超越纯文本生成的核心能力。

| 能力           | 核心方法                               | 作用                                                   |
| :------------- | :------------------------------------- | :----------------------------------------------------- |
| **工具调用**   | `model.bind_tools([tool_1, tool_2])`   | 让模型能“行动”，请求调用外部函数（如搜索、查数据库）。 |
| **结构化输出** | `model.with_structured_output(Schema)` | 强制模型按指定格式（如JSON）回复，便于程序解析。       |

**1. 工具调用示例**
```python
from langchain.tools import tool

# 定义工具
@tool
def get_weather(location: str) -> str:
    """获取指定地点的天气"""
    return f"{location}天气晴朗，72°F。"

# 绑定工具
model_with_tools = model.bind_tools([get_weather])

# 模型会智能决定是否调用工具
response = model_with_tools.invoke("波士顿天气怎么样？")
# response.tool_calls 包含工具调用请求
```

**2. 结构化输出示例**
```python
from pydantic import BaseModel, Field

# 定义期望的输出结构
class Movie(BaseModel):
    title: str = Field(description="电影标题")
    year: int = Field(description="发行年份")
    director: str = Field(description="导演")
    rating: float = Field(description="评分")

# 要求模型按此结构输出
model_with_structure = model.with_structured_output(Movie)
response = model_with_structure.invoke("提供《盗梦空间》的详情。")
# response 是一个已填充好的 Movie 实例
```

### 高级特性一览
Models 组件还封装了许多高级功能，让开发者能更精细地控制和监控模型行为。

| 特性              | 描述                                                         |
| :---------------- | :----------------------------------------------------------- |
| **多模态**        | 支持处理图像、音频等非文本数据。                             |
| **推理过程**      | 可获取模型得出最终答案前的中间思考步骤。                     |
| **提示缓存**      | 对重复使用的提示内容，减少延迟和成本。                       |
| **速率限制**      | 通过 `rate_limiter` 参数控制请求频率，避免触发提供商的限制。 |
| **Token用量追踪** | 轻松获取每次调用的Token消耗数据，用于成本监控。              |
| **可配置模型**    | 创建在运行时可通过配置切换提供商的动态模型，便于测试和切换。 |

### 总结与关联
LangChain 的 **Models** 组件提供了一个统一、强大且高度可配置的接口来与各种大语言模型交互。它是构建一切应用的基础，与之前讨论的 **Agents** 组件紧密相关：Agent 正是将 Model 作为其“推理引擎”，并利用 Model 的**工具调用**能力来与外部世界互动。

## Messages（消息）

根据你提供的 LangChain 官方文档，我来为你详细解析 **Messages（消息）** 这个核心组件。Messages 是 LangChain 中模型交互的**基本上下文单元**，它不仅仅是文本，更是一个结构化的信息载体，用于在模型、用户和工具之间传递状态。

### 核心构成：消息的三要素
每条消息都包含三个关键部分：
*   **角色**：标识消息的发送者类型（如 `system`、`user`、`assistant`、`tool`）。
*   **内容**：消息的实际负载，可以是纯文本，也可以是图像、音频等多模态数据。
*   **元数据**：可选的附加信息，如消息ID、Token用量、响应元数据等。

### 四大核心消息类型
理解四种基本消息类型，是构建对话流程的基础。

| 类型              | 角色        | 核心作用                                                     | 关键字段/说明                                                |
| :---------------- | :---------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **SystemMessage** | `system`    | **设定规则**：在对话开始时，为模型设定角色、行为和全局上下文。 | `content`: 包含具体指令的字符串。                            |
| **HumanMessage**  | `user`      | **代表用户**：承载用户的输入，可以是文本或多模态内容。       | `content`: 用户的问题或指令；可包含多模态数据块。            |
| **AIMessage**     | `assistant` | **模型回复**：模型的输出结果，可能包含文本、工具调用请求或元数据。 | `content`: 模型的文本回复；`tool_calls`: 工具调用请求列表；`usage_metadata`: Token用量信息。 |
| **ToolMessage**   | `tool`      | **传递结果**：将工具执行后的结果返回给模型。                 | `content`: 工具执行的字符串结果；`tool_call_id`: **必须**与对应 `AIMessage` 中工具调用的ID匹配。 |

### 消息的三种调用形式
你可以根据场景的复杂程度，选择最便捷的消息传递方式：

**1. 纯文本字符串（最简单）**
适合单轮、无上下文的独立请求。
```python
response = model.invoke("写一首关于春天的俳句。")
```

**2. 消息对象列表（最标准）**
适合多轮对话、需要系统指令或包含多模态内容的场景。
```python
from langchain.messages import SystemMessage, HumanMessage

messages = [
    SystemMessage("你是一位诗歌专家。"),
    HumanMessage("写一首关于春天的俳句。")
]
response = model.invoke(messages)
```

**3. 字典格式（兼容性强）**
直接使用与 OpenAI API 兼容的字典格式。
```python
messages = [
    {"role": "system", "content": "你是一位诗歌专家。"},
    {"role": "user", "content": "写一首关于春天的俳句。"}
]
response = model.invoke(messages)
```

### 深度理解：消息内容的标准化
为了统一处理不同模型提供商返回的多样化数据（如推理步骤、多模态数据），LangChain 引入了 **标准内容块** 的概念。通过访问消息的 `content_blocks` 属性，你可以获得一个类型安全、结构统一的内容列表，极大简化了跨模型的数据处理逻辑。
```python
# 假设 response 是从某模型获取的 AIMessage
for block in response.content_blocks:
    if block["type"] == "text":
        print(f"文本: {block['text']}")
    elif block["type"] == "reasoning":
        print(f"推理: {block['reasoning']}")
    elif block["type"] == "tool_call":
        print(f"工具调用: {block['name']}, 参数: {block['args']}")
```

### 总结与关联
Messages 组件是 LangChain 数据流的核心。它将模型、用户和工具的交互统一为一种结构化的序列。理解它，你就理解了 LangChain 应用的“通用语言”。

## Tools（工具）

根据你提供的 LangChain 官方文档，我来为你详细解析 **Tools（工具）** 这个核心组件。简单来说，**工具是赋予 Agent 行动能力的关键**，它们让模型能够突破纯文本的限制，去获取实时数据、执行代码或操作外部系统。

### 如何创建一个工具
创建工具最直接的方法是使用 `@tool` 装饰器。工具的核心是：**函数的类型提示定义了输入模式，而文档字符串则成为模型决定何时调用它的“说明书”**。

```python
from langchain.tools import tool

@tool
def search_database(query: str, limit: int = 10) -> str:
    """搜索客户数据库中匹配查询的记录。

    Args:
        query: 要查找的搜索词
        limit: 返回结果的最大数量
    """
    # 这里是实际的数据库查询逻辑
    return f"找到 {limit} 条关于 '{query}' 的结果"
```

### 工具如何感知运行时环境
工具的强大之处在于它能在执行时感知和影响对话的上下文。通过为工具函数添加一个 **`runtime: ToolRuntime`** 参数，你可以解锁以下能力：

| 能力             | 通过 `runtime` 访问     | 核心作用与生命周期                                           |
| :--------------- | :---------------------- | :----------------------------------------------------------- |
| **短时记忆**     | `runtime.state`         | 读写当前对话的临时状态（如消息历史、临时变量），对话结束即消失。 |
| **不可变上下文** | `runtime.context`       | 读取在调用 Agent 时传入的、对话期间不变的配置（如 `user_id`）。 |
| **长时记忆**     | `runtime.store`         | 读写跨对话持久化的数据（如用户偏好、知识库），数据可跨会话存在。 |
| **流式反馈**     | `runtime.stream_writer` | 在工具执行过程中，向前端实时发送自定义的进度或状态消息。     |

**典型示例：结合短时与长时记忆**
```python
from langchain.tools import tool, ToolRuntime

@tool
def save_user_info(user_id: str, user_info: dict, runtime: ToolRuntime) -> str:
    """保存用户信息到长期存储，并更新当前会话状态。"""
    # 1. 写入长时记忆 (Store)，数据将持久化
    store = runtime.store
    store.put(("users",), user_id, user_info)

    # 2. 更新短时记忆 (State)，本次对话后续可直接使用
    return Command(
        update={
            "user_preferences": user_info.get("preferences", {}),
            "messages": [ToolMessage(
                content=f"已为用户 {user_id} 保存信息。",
                tool_call_id=runtime.tool_call_id
            )]
        }
    )
```

### 工具的返回与状态更新
工具的返回值类型决定了 Agent 的后续行为：

| 返回类型                 | 行为                                                         | 适用场景                                           |
| :----------------------- | :----------------------------------------------------------- | :------------------------------------------------- |
| **`str` (字符串)**       | 自动包装为 `ToolMessage`，模型直接阅读此结果并决定下一步。   | 大部分信息检索类工具。                             |
| **`dict` (字典)**        | 序列化后作为 `ToolMessage` 内容，模型可解析其中的结构化字段。 | 需要模型基于结构化数据做进一步推理时。             |
| **`Command` (命令对象)** | **同时更新 Agent 状态**，并可选择性地发送 `ToolMessage`。    | 工具执行后需要修改对话全局状态（如设置用户语言）。 |

### 预构建工具与 ToolNode
LangChain 提供大量开箱即用的预构建工具，同时，`ToolNode` 是在更底层的 LangGraph 工作流中执行工具的标准节点，它自动处理了并行执行、错误处理等复杂逻辑。

## Short-term Memory（短期记忆）

根据你提供的 LangChain 官方文档，我来为你详细解析 **Short-term Memory（短期记忆）** 组件。简单来说，短期记忆是让 Agent 在**单个会话线程（Thread）内**记住并管理对话历史的能力，是实现连贯多轮对话的基础。

### 核心机制：基于检查点的状态持久化
短期记忆的实现核心在于 **Checkpointer（检查点机制）**。它会将 Agent 的状态（默认是包含 `messages` 列表的 `AgentState`）持久化存储。每次与 Agent 交互时，相关的状态都会被保存，并在同一线程的下次调用时被自动加载，从而“记住”之前的对话。

```python
from langchain.agents import create_agent
from langgraph.checkpoint.memory import InMemorySaver

# 1. 创建一个检查点来保存状态
agent = create_agent(
    "gpt-5",
    tools=[...],
    checkpointer=InMemorySaver(),  # 开发时使用内存存储
)

# 2. 通过 thread_id 区分不同会话
config = {"configurable": {"thread_id": "1"}}
agent.invoke({"messages": [{"role": "user", "content": "我叫Bob。"}]}, config)
# 下次调用时，Agent会记得"Bob"
agent.invoke({"messages": [{"role": "user", "content": "我叫什么名字？"}]}, config)
```

### 生产环境配置
生产环境中，应使用数据库支持的检查点来保证数据持久性，例如 PostgreSQL：
```python
from langgraph.checkpoint.postgres import PostgresSaver

DB_URI = "postgresql://..."
with PostgresSaver.from_conn_string(DB_URI) as checkpointer:
    checkpointer.setup()  # 自动创建表结构
    agent = create_agent("gpt-5", tools=[...], checkpointer=checkpointer)
```

### 管理上下文窗口的三大策略
随着对话增长，完整的历史记录可能超出模型上下文窗口。文档提供了三种核心策略来管理消息历史，它们通常通过**中间件**实现。

| 策略         | 核心方法                                                     | 优点                                                         | 缺点与注意事项                                               |
| :----------- | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **修剪消息** | 使用 `@before_model` 中间件，基于令牌数或消息数量，保留最近的消息。 | 简单直接，能有效控制上下文长度。                             | 会**永久丢失**被修剪掉的信息，可能丢失重要的早期对话背景。   |
| **删除消息** | 使用 `@after_model` 或 `@before_model` 中间件，通过 `RemoveMessage` 删除特定消息。 | 可精细控制删除特定消息（如包含敏感词或过时的信息）。         | 需要小心维护对话结构的完整性（例如，确保工具调用和结果成对出现）。 |
| **总结消息** | 使用内置的 `SummarizationMiddleware`，用另一个模型将旧消息总结为一段摘要。 | **既保留关键信息，又释放上下文空间**，是更高级和推荐的方法。 | 需要额外调用模型进行总结，会增加一点延迟和成本。             |

### 如何在各处访问和修改记忆
短期记忆（即 `AgentState`）可以在 Agent 运行流程的多个环节被访问或修改：

| 访问位置         | 如何访问/修改                                                | 典型用途                                                     |
| :--------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **在工具中读取** | 通过 `ToolRuntime` 参数访问 `runtime.state`。                | 工具执行时获取当前对话状态（如用户ID、上一步结果）。         |
| **在工具中写入** | 工具返回一个 `Command` 对象，其中 `update` 字段包含要更新的状态。 | 工具执行后，将结果或提取的信息持久化到状态中，供后续使用。   |
| **在提示词中**   | 通过 `@dynamic_prompt` 中间件，从 `request.runtime.context` 或 `state` 获取信息来构建提示。 | 根据对话历史或用户偏好动态生成个性化的系统或用户提示。       |
| **在模型调用前** | 使用 `@before_model` 中间件，直接处理 `state` 中的消息列表。 | 实现消息修剪、删除或注入额外的上下文信息。                   |
| **在模型调用后** | 使用 `@after_model` 中间件，处理模型生成的回复。             | 对模型回复进行后处理，如过滤敏感词、验证格式或触发状态更新。 |

### 总结
短期记忆是构建连贯对话体验的基石。LangChain 通过统一的 `AgentState` 和灵活的 `Checkpointer` 机制管理它，并通过强大的**中间件**系统，让你能在 Agent 生命周期的关键节点上，对记忆的读取、修改和管理策略进行精细定制。

## Streaming（流式传输）

根据你提供的 LangChain 官方文档，我来为你详细解析 **Streaming（流式传输）** 这个核心组件。流式传输并非一个独立模块，而是一种贯穿 Agent 运行时的**数据推送机制**。它能让应用实时接收 Agent 内部的状态变化、思考过程和生成结果，是提升用户体验和实现复杂交互的关键。

### 核心价值：实时反馈
对于基于大语言模型的应用，延迟是影响体验的主要因素。Streaming 通过渐进式地输出内容，让用户能立刻看到响应开始生成，而不是对着空白屏幕等待，这极大地改善了感知性能和交互的流畅性。

### 三种核心流式模式
通过向 `stream()` 或 `astream()` 方法传入 `stream_mode` 参数，你可以精准控制想要接收的数据类型。

| 模式 (`stream_mode`) | 数据内容 (`chunk["data"]`)                                   | 主要用途与场景                                               |
| :------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **`"updates"`**      | Agent 状态更新。每次节点（如 `model` 或 `tools`）执行后，包含完整消息的 `AgentState` 字典。 | 展示 Agent 的执行步骤与进度，例如："正在思考..." → "正在调用工具..." → "正在生成回答..."。 |
| **`"messages"`**     | LLM 生成的令牌（Token）流。返回 `(token, metadata)` 元组，其中 `token` 是 `AIMessageChunk` 对象。 | **实现打字机效果**，逐字或逐块显示模型的回复文本或工具调用参数的生成过程。 |
| **`"custom"`**       | 用户自定义数据。通过在工具或节点内使用 `get_stream_writer()` 函数发送的任意数据。 | 发送工具内部的执行进度，例如："正在查询数据库..."、`"已获取10/100条记录"`。 |

### 实战示例：组合模式监听完整流程
下面的代码展示了如何同时使用 `updates` 和 `custom` 模式，来监听一个包含自定义进度反馈的 Agent。

```python
from langchain.agents import create_agent
from langgraph.config import get_stream_writer

# 1. 定义工具，内部发送自定义流式消息
def get_weather(city: str) -> str:
    """获取指定城市的天气。"""
    writer = get_stream_writer()
    writer(f"正在查询 {city} 的天气数据...")  # 发送自定义进度
    # ... 实际的API调用逻辑 ...
    writer(f"已成功获取 {city} 的天气数据。")
    return f"{city} 今天晴天。"

agent = create_agent(model="gpt-5-nano", tools=[get_weather])

# 2. 同时以 'updates' 和 'custom' 模式监听
for chunk in agent.stream(
    {"messages": [{"role": "user", "content": "旧金山天气如何？"}]},
    stream_mode=["updates", "custom"],
    version="v2", # 推荐使用统一的 v2 格式
):
    mode = chunk["type"] # 'updates' 或 'custom'
    data = chunk["data"]
    
    if mode == "custom":
        print(f"[进度] {data}")
    elif mode == "updates":
        # data 是一个字典，键为节点名（如 'model', 'tools'）
        for step, update in data.items():
            if step == "model":
                last_msg = update["messages"][-1]
                # 检查是否为工具调用请求
                if hasattr(last_msg, 'tool_calls') and last_msg.tool_calls:
                    print(f"[行动] 模型请求调用工具: {last_msg.tool_calls[0]['name']}")
            elif step == "tools":
                last_msg = update["messages"][-1]
                print(f"[结果] 工具执行完毕: {last_msg.content}")
```

**输出效果预览：**
```
[行动] 模型请求调用工具: get_weather
[进度] 正在查询 San Francisco 的天气数据...
[进度] 已成功获取 San Francisco 的天气数据。
[结果] 工具执行完毕: San Francisco 今天晴天。
[最终] 模型正在生成回复...
```

### 关键高级特性
文档还强调了一些重要的高级用法：

*   **流式传输推理过程**：对于支持推理的模型（如 Claude），可以通过过滤 `content_blocks` 中 `type` 为 `"reasoning"` 的块，来实时显示模型的“思考过程”。
*   **流式处理子 Agent**：在多 Agent 系统中，为每个 Agent 设置 `name` 属性，并在 `stream` 中开启 `subgraphs=True`，即可在 `metadata` 中通过 `lc_agent_name` 区分数据来自哪个 Agent。
*   **v2 流式格式**：推荐使用 `version="v2"` 参数，它将所有输出统一为 `{"type": "...", "data": ...}` 的字典结构，处理起来更简洁、一致。

Streaming 是构建响应式、可观测 AI 应用的核心技术。通过组合不同的流式模式，你可以精细控制前端展示，为用户提供从执行进度到思考过程的全方位实时反馈。

## Structured Output（结构化输出）

根据你提供的 LangChain 官方文档，我来为你详细解析 **Structured Output（结构化输出）** 这个重要特性。简单来说，它能让 Agent 的回复不再是难以解析的自然语言，而是按照你预先定义的“格式”（Schema）返回的、可直接被程序使用的**结构化数据**。

### 核心价值：从“解析文本”到“使用数据”
在没有结构化输出时，你需要编写复杂且脆弱的正则表达式或提示词工程来从模型回复中提取信息。而使用结构化输出后，Agent 可以直接返回一个标准的 JSON 对象、一个 Pydantic 模型实例或一个数据类，让下游代码处理起来准确、高效且可靠。

### 两种实现策略
`create_agent` 通过 `response_format` 参数支持两种底层策略。通常你只需传入一个 Schema 类型，LangChain 会自动根据模型能力选择最佳策略。

| 策略                 | 实现原理                                                     | 适用场景                                             |
| :------------------- | :----------------------------------------------------------- | :--------------------------------------------------- |
| **ProviderStrategy** | 使用模型提供商（如OpenAI、Anthropic）的原生API功能，由服务端直接保证输出格式。 | **首选方案**，当模型提供商原生支持时，可靠性最高。   |
| **ToolStrategy**     | 将 Schema 包装成一个“虚拟工具”，模型通过“调用”这个工具来输出结构化数据。 | **通用后备方案**，适用于所有支持工具调用的现代模型。 |

### 基础用法：自动选择策略
最简单的用法是直接将你的 Pydantic 模型、数据类或 TypedDict 传给 `response_format`。LangChain 会自动检测模型能力并选择最佳策略。

```python
from pydantic import BaseModel, Field
from langchain.agents import create_agent

# 1. 定义期望的输出结构
class ContactInfo(BaseModel):
    """提取联系人信息"""
    name: str = Field(description="姓名")
    email: str = Field(description="邮箱地址")
    phone: str = Field(description="电话号码")

# 2. 创建 Agent 时指定输出格式
agent = create_agent(
    model="gpt-5",
    response_format=ContactInfo  # 自动选择 ProviderStrategy 或 ToolStrategy
)

# 3. 执行并获取结构化结果
result = agent.invoke({
    "messages": [{"role": "user", "content": "从文本提取信息：张三, zhangsan@email.com, 13800138000"}]
})

# 4. 直接从 state 中取得已解析好的数据
contact = result["structured_response"]
print(f"姓名: {contact.name}, 邮箱: {contact.email}")
# 输出: 姓名: 张三, 邮箱: zhangsan@email.com
```

### 进阶配置：精细控制 ToolStrategy
当使用 `ToolStrategy` 时，你可以通过以下参数进行更精细的配置：

| 参数                       | 描述                                                         | 示例                                            |
| :------------------------- | :----------------------------------------------------------- | :---------------------------------------------- |
| **`schema`**               | **(必需)** 定义输出结构，支持 Pydantic、数据类、TypedDict 或 JSON Schema。 | `schema=ContactInfo`                            |
| **`tool_message_content`** | 自定义在对话历史中，工具成功执行后留下的消息内容。           | `tool_message_content="已成功捕获联系人信息！"` |
| **`handle_errors`**        | 配置当模型输出不符合 Schema 时的错误处理策略。这是保证鲁棒性的关键。 | `handle_errors="请提供一个有效的邮箱地址。"`    |

### 强大的错误处理机制
模型可能出错，`ToolStrategy` 提供了智能的重试机制。通过 `handle_errors` 参数，你可以定义遇到错误时的行为。

```python
from langchain.agents.structured_output import ToolStrategy

# 策略1：使用自定义字符串提示模型重试
response_format = ToolStrategy(
    schema=ContactInfo,
    handle_errors="格式错误，请确保姓名、邮箱、电话都填写完整。"
)

# 策略2：定义一个错误处理函数，根据异常类型返回不同提示
def custom_handler(error: Exception) -> str:
    if isinstance(error, StructuredOutputValidationError):
        return "数据验证失败，请检查字段是否符合要求。"
    return "发生未知错误，请重试。"

response_format = ToolStrategy(
    schema=ContactInfo,
    handle_errors=custom_handler
)
```
当输出格式错误时，框架会自动将错误信息反馈给模型，并引导它进行修正，从而大大提高了最终输出的成功率。

### 总结
Structured Output 将 Agent 从一个“会说话的模型”升级为一个“可编程的、返回可靠数据的 API 端点”。它是构建生产级 AI 应用不可或缺的一环，确保了系统间数据交换的稳定性和准确性。
